1. Objective

Build a Transformer model to generate Bach-style cello music from MIDI files. The model learns to predict the next note and duration given a sequence of previous tokens, then generates new music autoregressively from a seed token.

2. Data Pipeline

2.1 MIDI Parsing & Tokenization

Each MIDI file (Bach chorales/cello suites) is parsed into two parallel token streams:

  • Note tokens: pitch names with octave (e.g., C4, G3), rests, time signatures (3/4TS), key signatures (G:major), and a START marker.
  • Duration tokens: quarter-note lengths as strings (e.g., 1.0, 0.5, 0.25).

Overlapping sequences of length 51 are created (50 input tokens + 1 target), following the reference implementation's sliding window approach.

2.2 Dataset

36 Bach MIDI files from the music21 corpus are used, yielding thousands of training sequences. Each sequence is tokenized into integer indices using separate vocabularies for notes and durations.

3. Model Architecture

Note Tokens ──→ [Embedding + Sinusoidal Pos Enc] ──┐
                                                     ├──→ Concatenate ──→ Transformer Block ──→ Note Head
Duration Tokens → [Embedding + Sinusoidal Pos Enc] ──┘                                    └──→ Duration Head
ComponentDetails
Note Embedding128-dim token embedding + sinusoidal positional encoding
Duration Embedding128-dim token embedding + sinusoidal positional encoding
Concatenated Dim256 (128 + 128)
Transformer Block4-head causal self-attention + FFN (256-dim)
Note Output HeadLinear → softmax over note vocabulary
Duration Output HeadLinear → softmax over duration vocabulary
Total Parameters~450K

Key Design Choices

  • Causal masking prevents the model from seeing future tokens, enabling autoregressive generation.
  • Dual output heads predict note and duration simultaneously from shared attention representations.
  • Sinusoidal positional encoding (not learned) generalizes to different sequence lengths.

5. Generation Process

Music is generated autoregressively:

  1. Seed the model with a START token (note) and 0.0 (duration).
  2. At each step, the model predicts probability distributions over the next note and duration.
  3. Sample from these distributions using temperature-controlled sampling.
  4. Append the sampled tokens and repeat until reaching the desired length.
  5. Convert the token sequence back to music21 objects and export as MIDI.
Temperature controls output diversity:
T=0.3: Very conservative, repetitive but "safe" melodies
T=0.5: Balanced — coherent structure with some variety
T=0.8: More creative, occasional surprising note choices
T=1.0: Maximum variety, may produce less coherent sequences

6. Results

6.1 Training

The training loss decreases steadily over 200 epochs, indicating the model learns the statistical patterns of Bach's music — common note progressions, rhythmic patterns, and key relationships. Loss curves and sample MIDI files are generated periodically during training.

6.2 Generated Music

The model generates MIDI files at four different temperatures. At lower temperatures (0.3–0.5), the output tends to follow common patterns from the training data — scale passages, arpeggios, and familiar rhythmic figures. At higher temperatures (0.8–1.0), the output is more exploratory with wider interval leaps and varied rhythms.

Impact of Differences

  • More training data (36 vs 6 files): The model sees more varied Bach compositions, leading to a richer vocabulary and more diverse generation patterns.
  • Fewer epochs (200 vs 5000): With more data and PyTorch's efficient training, fewer epochs suffice. The model converges faster per epoch with more sequences.
  • Temperature comparison: Multiple temperature outputs let us directly observe the diversity-coherence tradeoff.
  • 4 vs 5 attention heads: Minimal impact on quality — the key factor is total embedding dimension, which remains 256.
